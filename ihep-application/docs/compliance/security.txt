        <h2>1. Security Overview</h2>
        <p>IHEP operates a <span class="highlight">compliance-first, zero-trust security architecture</span> designed to meet the most stringent healthcare data protection requirements. Our security framework is built on three foundational pillars: prevention, detection, and recovery.</p>

        <div class="feature-box">
            <h3 style="margin-top: 0;">Security Philosophy</h3>
            <p><strong>Trust Nothing. Verify Everything.</strong> IHEP implements zero-trust principles where all users, devices, and network traffic are continuously authenticated, authorized, and encrypted—regardless of location or previous trust status.</p>
        </div>

        <h2>3. Encryption and Data Protection</h2>

        <h3>3.1 Data at Rest</h3>
        <p>All Protected Health Information stored in IHEP systems is encrypted using:</p>
        <ul>
            <li><span class="highlight">AES-256 encryption</span> (Advanced Encryption Standard, 256-bit key)</li>
            <li>Industry-standard encryption algorithms validated by NIST</li>
            <li>Key management through Google Cloud KMS (Key Management Service)</li>
            <li>Automatic key rotation every 90 days</li>
            <li>Secure key storage with HSM (Hardware Security Module) backing</li>
        </ul>

        <h3>3.2 Data in Transit</h3>
        <p>All data transmitted to and from IHEP systems is encrypted using:</p>
        <ul>
            <li><span class="highlight">TLS 1.3</span> (Transport Layer Security 1.3) for all network communications</li>
            <li>Perfect Forward Secrecy (PFS) for ephemeral session keys</li>
            <li>Certificate pinning to prevent man-in-the-middle attacks</li>
            <li>OCSP stapling for certificate status validation</li>
            <li>Minimum 2048-bit RSA keys for certificate signing</li>
        </ul>

        <h2>4. Authentication and Access Control</h2>

        <h3>4.1 Multi-Factor Authentication (MFA)</h3>
        <p>All user access to IHEP systems requires:</p>
        <ul>
            <li>Primary factor: Username and strong password (minimum 12 characters, complexity requirements)</li>
            <li>Secondary factor: Time-based one-time password (TOTP) or hardware security key</li>
            <li>Biometric authentication available on supported devices</li>
            <li>Risk-based authentication for unusual access patterns</li>
        </ul>

        <h3>4.2 Role-Based Access Control (RBAC)</h3>
        <p>Access to PHI and sensitive systems is strictly controlled using:</p>
        <ul>
            <li>Principle of least privilege—users granted minimum necessary permissions</li>
            <li>Role definitions aligned with job functions and security levels</li>
            <li>Regular access reviews and justification of elevated permissions</li>
            <li>Automated access removal upon employment termination</li>
            <li>Segregation of duties to prevent unauthorized transactions</li>
        </ul>

        <h3>4.3 Session Management</h3>
        <ul>
            <li>Automatic session timeout after 30 minutes of inactivity</li>
            <li>Session encryption and secure cookie handling</li>
            <li>Concurrent session limits to prevent account sharing</li>
            <li>Immediate revocation capability for compromised sessions</li>
        </ul>

        <h2>5. Network Security</h2>

        <h3>5.1 Architecture</h3>
        <p>IHEP network is protected through:</p>
        <ul>
            <li>VPC (Virtual Private Cloud) isolation with public/private subnets</li>
            <li>Web Application Firewall (WAF) blocking malicious traffic patterns</li>
            <li>DDoS protection and mitigation</li>
            <li>VPN requirements for administrative access</li>
            <li>Network segmentation isolating PHI from non-critical systems</li>
        </ul>

        <h3>5.2 Intrusion Detection and Prevention</h3>
        <ul>
            <li>Real-time network monitoring with automated threat detection</li>
            <li>Intrusion Prevention System (IPS) to block known attack patterns</li>
            <li>Behavioral analysis detecting anomalous network activity</li>
            <li>Automated incident response triggering alerts and containment</li>
        </ul>

        <h2>6. Data Audit and Logging</h2>

        <h3>6.1 Comprehensive Audit Trails</h3>
        <p>IHEP maintains immutable audit logs for all PHI access and system events:</p>
        <ul>
            <li>User authentication attempts (successful and failed)</li>
            <li>All PHI access with timestamp, user ID, data accessed</li>
            <li>Modifications to patient data with before/after values</li>
            <li>Administrative actions and configuration changes</li>
            <li>Data exports and disclosures with recipient information</li>
            <li>Security events and policy violations</li>
        </ul>

        <h3>6.2 Log Management</h3>
        <ul>
            <li><span class="highlight">Blockchain-style cryptographic chaining</span> to ensure log integrity</li>
            <li>Centralized log aggregation with immutable storage</li>
            <li>Automated log analysis detecting suspicious patterns</li>
            <li>Retention: 7 years minimum per HIPAA requirements</li>
            <li>Export capabilities for incident investigation and audits</li>
        </ul>

        <h2>7. Vulnerability Management</h2>

        <h3>7.1 Scanning and Assessment</h3>
        <ul>
            <li>Automated vulnerability scanning: daily for infrastructure, weekly for applications</li>
            <li>Penetration testing: quarterly internal, annual external</li>
            <li>Software composition analysis identifying vulnerable dependencies</li>
            <li>Security code review of all source code changes before deployment</li>
        </ul>

        <h3>7.2 Patch Management</h3>
        <ul>
            <li>Critical patches: deployed within 48 hours</li>
            <li>High-priority patches: deployed within 1 week</li>
            <li>Standard patches: deployed within monthly maintenance windows</li>
            <li>Automatic rollback capability for failed patches</li>
        </ul>

        <h2>8. Incident Response</h2>

        <h3>8.1 Incident Detection and Response</h3>
        <ul>
            <li>24/7 Security Operations Center (SOC) monitoring</li>
            <li>Automated alerts for security events exceeding thresholds</li>
            <li>Incident classification and severity assessment within 15 minutes</li>
            <li>Containment actions initiated immediately for critical incidents</li>
            <li>Communication to affected parties within 24 hours for breaches</li>
        </ul>

        <h3>8.2 Investigation and Recovery</h3>
        <ul>
            <li>Forensic analysis of all security incidents</li>
            <li>Root cause analysis with corrective actions</li>
            <li>Evidence preservation for potential legal proceedings</li>
            <li>System recovery and data restoration procedures</li>
            <li>Post-incident review and lessons learned documentation</li>
        </ul>

        <h2>9. Business Continuity and Disaster Recovery</h2>

        <h3>9.1 Backup Strategy</h3>
        <ul>
            <li>Continuous incremental backups with hourly snapshots</li>
            <li>Daily full backups with 30-day retention</li>
            <li>Geographic redundancy with backups in multiple regions</li>
            <li>Automated backup integrity verification</li>
            <li>Recovery Time Objective (RTO): 1 hour</li>
            <li>Recovery Point Objective (RPO): 15 minutes</li>
        </ul>

        <h3>9.2 Disaster Recovery</h3>
        <ul>
            <li>Failover to secondary data center: automatic within 5 minutes</li>
            <li>Business Continuity Plan tested quarterly</li>
            <li>Redundant infrastructure across availability zones</li>
            <li>Service availability guarantee: 99.9% uptime SLA</li>
        </ul>

        <h2>10. Vendor and Third-Party Management</h2>

        <h3>10.1 Business Associate Agreements</h3>
        <p>All vendors handling PHI must execute Business Associate Agreements (BAAs) requiring:</p>
        <ul>
            <li>Compliance with HIPAA security and privacy rules</li>
            <li>Subcontractor agreements cascading security requirements</li>
            <li>Right to audit and security assessments</li>
            <li>Incident notification obligations</li>
            <li>Data destruction procedures upon contract termination</li>
        </ul>

        <h3>10.2 Vendor Assessment</h3>
        <ul>
            <li>Security assessment before vendor engagement</li>
            <li>Annual compliance verification</li>
            <li>Incident reporting requirements</li>
            <li>Right to audit vendor security practices</li>
        </ul>

        <h2>11. Staff Training and Security Awareness</h2>

        <h3>11.1 Onboarding</h3>
        <ul>
            <li>HIPAA compliance training (2 hours)</li>
            <li>Security awareness training (4 hours)</li>
            <li>Role-specific security training before system access</li>
            <li>Annual recertification of compliance understanding</li>
        </ul>

        <h3>11.2 Ongoing Education</h3>
        <ul>
            <li>Monthly security awareness communications</li>
            <li>Phishing simulation campaigns with education</li>
            <li>Breach response training and tabletop exercises</li>
            <li>Updates on emerging threats and vulnerabilities</li>
        </ul>
